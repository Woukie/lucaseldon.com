Federated Communications Platform

I want to create a communications platform matching the features of Discord whilst also meeting the following requirements

  • Users can host their own servers much like how team-speak works, significantly reducing hosting costs. However, due to usability, I will have to host small servers myself in order to support group chat and dm style communication
  • Browser indexing. With the rise of good SEO, search engines have become more useless than ever. By letting search engines index threads from configured servers, we can inject useful search results into the internet again. This feature also means that servers will have a public browser-accessible web-interface that doesn't require authentication to view
  • Merge group-chats and DMs into servers. Servers should be flexible enough to support many styles of communication through the way they are set up. In other words, users start with a 2 person server that acts like a dm, to ticking some boxes that change the UI to look like a server, to ticking some more boxes and becoming a fully-fledged internet-searchable community server
  • Democratized authentication, along-side our own auth server for account management. Authentication servers can be hosted by everyone, and server owners can whitelist specific auth servers. Even if we go down, the platform can live on
  • Free and open-source. If someone wants a feature, they can just make it
  • E2E encryption for small servers. Obviously this would be useless for public servers, but group chat and dm sized servers could use this
Here are some reasons why I haven't started this yet
  • Funding, audience. Unless I get these, this project will never happen
  • Manual server-hosting is not easy, nor is it geared towards the average user that just wants a few group chats. We'll likely have to host servers ourselves and have server creation happen automatically when a 'message' button is pressed. We would have to force users to host servers themselves (or charge them) once a specific user count is met, which is a bit of a turn-off
  • I haven't decided exactly how communication will function. All I know is that it will be real-time and thread-based. This approach provides us with maximum flexibility when designing a configuration system that supports various communication styles (e.g., channels, Reddit-like, forums, and threads). I haven't finalized the specific implementation details yet
  • No revenue model. I don't plan on having a revenue model immediately (except managed hosting). I may consider cosmetics down the line
  • Idea refinement, need to generate a specification that lays out the features beyond being 'Discord-like'. Moderation, UI, ease of use, role management etc..
